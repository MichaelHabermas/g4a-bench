Bundle Size Process Trace

This is a generalized evaluator trace pointed at bundle size. It is not a bundle-size product and it is not a final score.

Generalized Process

StepQuestionOutput Contract
criterion-decompositionWhat does the spec actually allow?Two valid target paths: 15% total production bundle reduction, or 20% initial-load reduction via code splitting.
evidence-discoveryWhere does this repo keep measurement evidence?Search for structured ledgers, before/after bundle JSON, delta reports, build logs, treemaps, and measurement scripts.
claim-extractionWhat path is the team claiming?Extract stated claims, but mark them self-reported until artifacts or reruns support them.
artifact-extractionCan raw before/after numbers be read from artifacts?Prefer machine-readable JSON. Markdown summaries are secondary. Copied specs are noise.
target-resolutionWhich allowed target path, if any, do the numbers satisfy?Evaluate total and initial-load paths independently instead of collapsing them into one bundle score.
comparability-checkAre before and after measurements comparable?Check same build command, same output scope, same compression basis, same app functionality, and no deletion-as-optimization.
independent-reproductionHas the harness reproduced the measurement?This prototype has not rerun builds. It can only mark artifact-backed, not independently verified.

Spec Anchor

Category 2: Bundle Size What you are measuring: The size of the production frontend bundle. Large bundles slow down initial page load, hurt performance on slow networks, and waste bandwidth. You are looking for oversized dependencies, missing code splitting, unused imports, and opportunities to reduce what the browser has to download. How to Measure • Build the production frontend and record the total output size • Use a bundle visualization tool (e.g., rollup-plugin-visualizer, vite-bundle-analyzer, or source-map-explorer) to generate a treemap of the bundle • Identify the largest chunks and the largest individual dependencies within them • Check for unused dependencies: cross-reference package.json dependencies against actual imports in the source code • Evaluate whether code splitting is in use and where lazy loading could reduce initial load Audit Deliverable Metric Your Baseline Total production bundle size ___ KB Largest chunk ___ (name + size) Number of chunks ___ Top 3 largest dependencies List with sizes Unused dependencies identified List Improvement Target 15% reduction in total production bundle size, or implement code splitting that reduces initial page load bundle by 20%. Provide before/after bundle analysis output. Removing functionality to shrink the bundle does not count.
